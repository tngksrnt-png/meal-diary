# ReNA Group HR Dashboard — 구현 내역 (재구현용 명세)

> 기준일: 2026-04-22
> 목적: 이 문서 하나만으로 **다른 AI / 다른 스택(Power Apps, Power BI, Dataverse 등)으로 동일 기능을 재구현**할 수 있도록 현재 구현된 모든 화면·데이터·규칙을 디테일하게 정리.
> 현재 스택: Next.js 16 (App Router) + TypeScript + Tailwind v4 + Motion + Zod + Supabase (Postgres + Auth + RLS) + recharts + xlsx

## 0. 도메인 개요

**조직 계층 (4단계)**
```
카테고리 (BU) = HQ / Sorting / MR / WtE / Shredding / Oil·Others
   └─ 법인 (companies, 17개)
        ├─ 사업장 (worksites) ─┐
        │                       ├─ 부서 (departments, 자체 parent 계층 가능)
        └─ 부서가 worksite_id 없이 법인 직속일 수도 있음 (HQ 형태)
```

- HQ 카테고리는 `is_hq=true` → 사업장 개념 없이 법인 직속 `부문(division)` → 하위 `팀/파트` 2단계.
- 일반 카테고리는 `사업장 → 부서` 2단계.
- 부서(`departments`)는 단일 테이블이며 `kind ∈ {division, team, part}` 로 구분.
- 직원(`employees`)은 `company_id`(필수) + `worksite_id`(nullable) + `department_id`(nullable) 를 가짐.

**직원 상태**: `재직 / 휴직 / 퇴직` 세 가지 상태(`status_code`)로만 관리.

**사용자**: 그룹 HR 단일 역할(`group_admin`). 개별 직원은 로그인 사용자가 아니라 **데이터**임.

## 1. 인증 · 권한 · 보안

### 1.1 로그인
- **Supabase Auth + Google OAuth** 단일 공급자.
- 엔트리 포인트: `/login` → Google 선택 → `/auth/callback`에서 code→session 교환 → `/`(대시보드).
- 로그아웃: `/auth/logout`에 POST → `supabase.auth.signOut()`.

### 1.2 이메일 화이트리스트 (이중 게이트)
- `profiles.email` 테이블이 허용 이메일 목록. 로그인한 `auth.users` 중 `profiles`에 이메일이 있는 계정만 진입 가능.
- **신규 가입 시 자동 연결**: `public.handle_new_user` PostgreSQL 트리거가 `auth.users` INSERT 후 실행, 이메일이 화이트리스트에 있으면 `profiles.user_id`를 해당 auth user id로 backfill.
- **이미 로그인했던 사용자를 나중에 화이트리스트에 추가하는 케이스**: `addWhitelistAction`은 Service Role로 `auth.users`를 조회해서 기존 user_id를 즉시 backfill (트리거는 신규 가입에만 발동).

### 1.3 접근 제어 (프론트 게이트)
- `src/lib/auth.ts`
  - `getSession()` — `supabase.auth.getUser()` 래핑
  - `getAllowedProfile()` — user.id로 `profiles.user_id` 매칭 행 조회
  - `requireAdmin()` — 비로그인 → `/login` 리다이렉트, 로그인했으나 미허용 → `/no-access` 리다이렉트
- **모든 `(dashboard)` 하위 라우트는 `layout.tsx`에서 `requireAdmin()` 호출**. 미허용자는 이 레이아웃에 진입 불가.

### 1.4 Row-Level Security (RLS)
- 8개 테이블 모두 RLS enabled.
- 정책: `is_group_admin()` SQL 함수가 현재 auth user의 profiles.role === 'group_admin' 여부 반환. 모든 read/write는 이 함수가 true여야 허용.
- 공개 접근 없음 (anon으로는 어떤 행도 읽을 수 없음).

### 1.5 Server-side 관리자 클라이언트
- `src/lib/supabase/admin.ts` — Service Role Key 사용. 서버에서만 `auth.users` 조회용으로 제한 사용.

## 3. 비즈니스 규칙 · 자동 계산 (`src/utils/aggregations.ts`, `format.ts`)

### 3.1 근속 & 총경력 (DB 저장 X, 항상 런타임 계산)
- `tenureMonths(hire, termination)` = (termination ?? 오늘 − hire) / 30.4375. hire가 null이면 null.
- `totalCareerYears(careerBefore, hire, termination)`
  - 입사전경력 있음: `careerBefore + tenureMonths/12`
  - 입사전경력 없음: `tenureMonths/12`만
  - 둘 다 null이면 null
- 표시 포맷 `formatTenureYears(months)`: 12개월 미만 → `"N개월"`, 이상 → `"X.Y년"`.
- **직원 폼에서 "총경력" 입력은 제거됨**. 입사일 + 입사전경력만 입력, 총경력은 자동.

### 3.2 만 나이 `ageFromBirth(birth, asOf=today)`
- 생일 전/후 보정 후 정수 반환. 0~110 범위 밖이면 null (오기재 방어).
- 퇴직 시 평균 연령은 `asOf=termination_date`로 호출.

### 3.3 경력 버킷 `CAREER_BUCKETS`
`["1년 미만", "1~3년", "3~5년", "5~10년", "10년 이상"]`
`careerBucket(years)`가 해당 키 반환. null이면 "미기재".

### 3.4 연령대 버킷 `AGE_BUCKETS`
`["20대 이하(<30)", "30대(30-39)", "40대(40-49)", "50대(50-59)", "60대 이상(≥60)"]`

### 3.5 채용 시점 경력 버킷 (HiringTab 전용)
`["신입 (0년)", "1년 미만(0<y<1)", "1~3년", "3~5년", "5~10년", "10년 이상"]`

### 3.6 이직률 (turnoverRate)
= 퇴직 / (재직 + 퇴직). 휴직자는 분모/분자 모두 제외.
임계값: **30% 초과 시 빨강** (KPI 카드 `tone="danger"`).

### 3.7 단기 퇴직률 (RetiredTab)
= (재직기간 < 12개월인 퇴직자) / (전체 퇴직자 중 유효 재직기간). 30% 초과 시 빨강.

### 3.8 성별 임금 격차 (CompTab)
`(남평균 − 여평균) / 남평균`. 절대값 20% 초과 시 노랑 경고.

### 3.9 90일 신규입사 / YTD 신규입사 / YTD 퇴사 / 순증감
- 90일 컷오프: `daysAgoIso(90)` = 오늘 − 90일 ISO 문자열. `hire_date >= cutoff`
- YTD: `hire_date >= "${올해}-01-01"` / `termination_date >= "${올해}-01-01"`
- 순증감 = YTD 입사 − YTD 퇴사. 음수=빨강, 양수=녹색.

### 3.10 포맷 함수 `src/utils/format.ts`
| 함수 | 예시 |
|---|---|
| `formatKRW(n)` | ≥1억 → "1.23억원", ≥1만 → "5,000만원", 그 외 "N원" |
| `formatCount(n)` | "N명" |
| `formatPercent(n, digits=1)` | 0.234 → "23.4%" |
| `formatYears(n, digits=1)` | "3.5년" |
| `formatMonths(n)` | "14.2개월" |
| `formatTenureYears(months)` | 12 미만 → "N개월", 이상 → "X.Y년" |

## 5. 화면별 상세 기능

### 5.1 메인 대시보드 `/` (`src/app/(dashboard)/page.tsx`)

**URL 파라미터** (ScopeFilter + ScopeTabs가 수정):
- `bu` (카테고리 id)
- `company` (법인 id)
- `worksite` (사업장 id)
- `tab` ∈ `active|retired|worksites|comp|hiring` (default: `active`)

**스코프 우선순위**: `worksite > company > bu > 전사`.

### 5.6 Excel 업로드 `/admin/import` (`ImportClient`)

**양식 다운로드**: `/api/template?type=active|retired` — 헤더 행 + 샘플 1행 + "입력 참고" 시트(현재 드롭다운/사업장/부서 전체) 포함 .xlsx 생성.

**업로드 플로우**
1. 파일 선택 (.xlsx)
2. 시트 타입 라디오: 재직자 / 퇴직자
3. 기본 법인 드롭다운 (행에 `법인` 컬럼이 비어있을 때 fallback)
4. [가져오기] → `importEmployeesAction` 서버 액션
5. 결과 패널 표시: 저장된 건수, 스킵 건수, 법인별 집계, 스킵 사유별 집계

**시트 인식 로직** (`src/lib/excel-import.ts`)
- 워크북의 모든 시트를 순회하며 헤더 행에 `[사번, 이름, 입사일]` 모두 포함 확인
- 재직자: `퇴사일` 컬럼이 **없어야** 함. 퇴직자: **있어야** 함.
- 시트 명칭 무관, 헤더로 식별.

**컬럼 매핑 (재직자 양식)**
```
사번 · 이름 · 부서 · 직급 · 생년월일 · 입사일 · 성별 · 고용형태 ·
구분(내/외국인) · 국적 · 사업장 · 회계구분 · 직군 · 연봉(원) ·
채용경로 · 최종학력 · 입사전경력(년) · 재직상태 · 비고
```
**퇴직자 양식** = 재직자 + `퇴사일`, `퇴직사유`.
**선택 컬럼**: `법인` (있으면 행 단위 라우팅).

**행 단위 라우팅**
1. 행의 `법인`이 있으면 → `companyIdByName` 매칭. 매칭 실패 → 스킵 (사유 기록)
2. 없으면 → 업로드 폼의 `defaultCompanyId` 사용
3. 둘 다 없으면 → 스킵

**룩업 코드 검증**
- 각 코드 필드(`rank, employment_type, hire_channel, education, accounting_type, job_family, termination_reason`)는 해당 type의 lookups 집합에 code가 있을 때만 저장, 아니면 null.
- `gender`는 `"남"|"여"` 만, 외 null.
- `nationality_type`은 `"내국인"|"외국인"` 만, 외 null.

**사업장 / 부서 매칭**
- `worksiteIdByCoName[companyId::name]` / `departmentIdByCoName[companyId::name]` 맵 사용 (이름이 법인 간 중복이라도 법인 내에서 유일하다고 가정).

**업서트**
- `onConflict: "company_id,employee_no"` 로 upsert. `employee_no` 비어있는 행은 그대로 insert (중복 방지 안 됨, 입력 측 책임).

**`import_logs` 기록**
- 모든 행이 단일 법인이면 `company_id=그 법인`, 다수면 `null`.
- `upserted_count, skipped_count, uploaded_by, file_name, sheet_type, uploaded_at=now()`.

**revalidate**: `/`, `/org`, `/companies`, 영향받은 각 `/companies/[id]`.

**업로드 이력 섹션**: 최근 20건, 시간 / 법인 / 시트 타입 / 파일명 / "저장/스킵" 열.

### 5.7 조직 구조 관리 `/admin/org` (`OrgAdminClient`)

트리 뷰 + 인라인 CRUD.

**지원 액션 (노드 레벨별)**
| 노드 | 가능 작업 |
|---|---|
| 카테고리 | 생성 / 이름·code 편집 / ▲▼ 순서 / 삭제(소속 법인 있으면 차단) |
| 법인 | 생성(카테고리 선택) / 이름·대표자 편집 / ▲▼ / ↗ 다른 BU로 이동 / 삭제(사업장 OR 직원 있으면 차단) |
| 사업장 | 생성 / 이름 편집 / ▲▼ / ↗ 다른 법인으로 이동 (소속 직원·부서 cascade) / 삭제(부서 OR 직원 있으면 차단) |
| 부서 | 생성(worksite 또는 parent_department 선택) / 이름+kind 편집 / ▲▼ / ↗ 다른 사업장 or 부모 부서로 이동 (직원 cascade) / 삭제(하위 부서 OR 직원 있으면 차단) |

**순서 변경 (▲▼)**
- 같은 부모 집합 안에서 인접 노드와 `order_idx` swap.
- 임시값 -1로 두 번 update (DB atomic swap 대용).
- 첫/마지막 노드는 버튼 disabled.

**이동 (↗ 드롭다운)**
- 법인 이동 → `category_id` 변경만
- 사업장 이동 → 자기 법인뿐 아니라 **소속 직원·부서의 company_id 일괄 갱신**
- 부서 이동 → (새 사업장 또는 새 부모 부서) + 법인이 바뀌면 직원도 새 법인 + 새 worksite_id로 갱신

**검색**: 이름/code 부분일치 → 매칭 부모 자동 펼침 + 배경 강조.

**인라인 편집 UX**
- 행 더블클릭 = 인라인 수정 모드
- 신규 추가 입력: Enter 저장 / Esc 취소, autoFocus
- 에러는 빨간 배경 띠로 표시
- 평소 작업 버튼은 50% opacity, 행 hover 시 진하게.
- 부서 편집 시 `kind` 셀렉트(부문/팀/파트).

### 5.8 드롭다운 소스 관리 `/admin/settings` (`LookupsClient`)

8개 탭 (rank, employment_type, hire_channel, education, employee_status, termination_reason, accounting_type, job_family).

탭별 테이블:
| 컬럼 | 편집 |
|---|---|
| code | 생성 시만 (수정 불가) |
| label | 인라인 수정 가능 |

**사용 중 여부 판정**: 각 type에 대응하는 employees 컬럼(`rank → rank_code` 등)에 해당 code가 존재하면 삭제 차단.

### 5.9 허용 이메일 `/admin/users` (`WhitelistClient`)

- 하단 4블록: 고용형태(재직) · 직군(재직) · 퇴직유형 · 퇴직 평균 재직

**Collapsible 매트릭스 전체 지표**
- 행 = 지표 21개 (재직자현황 6 + 퇴직자현황 6 + 고용형태재직 4 + 직군현황 4) + 그룹 라벨
- 열 = "전체" + 각 비교 대상

## 7. Server Actions 카탈로그

### 7.1 `src/actions/employees.ts`
- `createEmployeeAction(input)` — Zod 검증, `company_id` 필수, 나머지 optional. `revalidatePath` `/`, `/org`, `/employees`, `/companies/[id]`.
- `updateEmployeeAction(input)` — id + 수정 가능한 모든 필드 (partial).
- `terminateEmployeeAction({id, termination_date, termination_reason_code, memo?})` — status→'퇴직'. **검증**: 이미 퇴직이면 거부, `termination_date < hire_date`면 거부.
- `reinstateEmployeeAction({id})` — 퇴직 → '재직' (termination_date/reason null).
- `setOnLeaveAction({id, onLeave})` — 재직↔휴직 토글. 퇴직 상태에서는 거부.
- `deleteEmployeeAction(id)` — 물리 삭제.

### 7.2 `src/actions/import.ts`
- `importEmployeesAction({sheetType, fileName, fileBase64, defaultCompanyId?})` — 5.6 전체 로직. base64로 받아 서버에서 xlsx 파싱.

### 7.3 `src/actions/org.ts`
CRUD + reorder + move (5.7 참고). 모든 action은 `requireAdmin()` 후 `revalidate()` (루트, /org, /companies, /admin/org).

### 7.4 `src/actions/lookups.ts`
- `createLookupAction, updateLookupAction(label+is_active), reorderLookupAction({id, direction}), deleteLookupAction(id)` — 사용 중이면 삭제 차단.

### 7.5 `src/actions/users.ts`
- `addWhitelistAction({email})` — Zod email 검증. Service Role로 `auth.users` 조회 후 기존 user_id backfill.
- `removeWhitelistAction(id)` — 본인 계정 삭제 차단.

## 11. E2E 시나리오 (동작 기준)

1. **로그인 성공** — 화이트리스트 이메일로 Google 로그인 → `/` 진입. profiles 행에 user_id 자동 연결.
2. **로그인 차단** — 비허용 이메일 → `/no-access`.
3. **Excel 업로드** — `/admin/import`에서 재직자 xlsx 업로드 → 결과 패널에 저장건수/법인별 집계/스킵사유. `/`, `/org`, `/companies/[id]` 수치 반영.
4. **직원 추가** — `/employees/new` → 저장 → `/employees/[id]` 이동 → `/employees` 리스트에서 조회 가능.
5. **퇴직 처리** — `/employees/[id]`에서 [퇴직] → 퇴사일·사유 입력 → 상태 `퇴직`. `/`의 퇴직 KPI +1, 재직 −1, 이직률 재계산.
6. **스코프 필터** — `/`에서 ScopeFilter로 Sorting 선택 → 모든 KPI/패널/탭이 Sorting 산하만 집계.
7. **드릴다운** — 대시보드 KPI "재직" 클릭 → `/employees?status=재직` (스코프 유지).
8. **조직 구조 변경** — `/admin/org`에서 사업장을 다른 법인으로 이동 → 소속 직원/부서도 법인이 자동 바뀜. 대시보드 수치 즉시 반영.
9. **드롭다운 편집** — `/admin/settings`에서 직급 추가 → 직원 폼/Excel 파서에서 즉시 사용 가능. 이미 사용 중인 직급은 삭제 불가 (비활성화만 가능).
10. **화이트리스트 추가** — 이미 Google 로그인 시도했던 이메일을 `/admin/users`에 추가 → `user_id` 자동 backfill → 당사자가 다시 로그인하면 즉시 진입.
